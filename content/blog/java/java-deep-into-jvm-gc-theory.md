简介

前面一段时间介绍了Java虚拟姐的基本概念，主要是介绍了Java虚拟机的内存布局以及在内存中的对象。围绕这一些内容，阐明了一些虚拟机内存分配以可能出现的异常，也初步提到了垃圾收集器。这一节重点来说明垃圾收集器的 实现以及一些理论上的分析。

垃圾收集器（Garbage Collection GC），它进行垃圾收集的主要工作内容就是需要对虚拟机对象是否是垃圾进行以及是否需要进行回收进行判定。

## 怎样才能算做垃圾?

怎么样才能算真正的垃圾呢？垃圾一定是没有利用价值的对象，在Java虚拟机里面实现了许多垃圾回收的算法，不同算法对垃圾的判定也不一致。一般来说，没有对象用到的对象(引用计数为0)和没有与重要对象形成关联的对象都可以判定为垃圾。即使如果能够判定出垃圾，那么又怎么进行回收呢?

### 1.引用计数法

引用计数法是判断Java内存对象是否死亡的最简单最快最粗糙的方法。Java对象都有个引用计数，它位于对象的对象头中，用来记录这个对象的应用次数。通常，如果该对象的引用次数为0，那么他就是垃圾了。

引用计数法也有一个问题就是循环引用的问题。对于现在的软件体系，尤其是一些企业级框架的出现，使得对象和对象之前的引用变成一个非常复杂的网状图的结构。如果有循环依赖的话，那么要释放当前对象，就需要释放引用对象，如此一来，就无限循环而不能结束。

所以在如此复杂的情况下，很难用引用计数法，故这种方法在主流的Java虚拟机里面是没有被采用的。

> 循环引用的问题可通过 Recycler 算法解决，但是在多线程环境下，引用计数变更也要进行昂贵的同步操作，性能较低，因此引用计数法是可以处理循环引用问题的。

### 2.可达性分析

可达性分析就是通过一系列的根对象（GC Roots）作为起始节点，从这些根节点向下搜索，有被引用到的对象就是存活的。

那么，什么样的条件的对象才能成为GC Roots的一系列的对象呢？

首先，虚拟机是使用c语言实现的，为了实现这一机制，虚拟机内部实现有很多对象，包括基本的数据类型，异常对象，类加载器等。

还有就是静态属性引用的对象以及本地方法栈中引用的对象，这些对象通常是开发系统中常用的变量以及连接其他程序的方法所要使用到的白能量。

虚拟机栈中的对象也是gc roots中的对象，这些对象通常控制线程调用的局部变量，临时变量等。

还有就是方法区常量引用的对象，也是gc roots的对象。

### 缓刑？

那些在可达性算法中被标记为不可达的对象，他们也并非必死无疑。在虚拟机的中的对象如果被标记为不可达，那么虚拟机要执行其finalize方法。但虚拟机并不一定保证finalize方法执行完毕。如果对象没有继承finalize方法或者finalize方法已经被执行，那么该对象就应该被真正的回收，除非该对象在finalize方法里面重新将自己与GC Roots上的对象进行关联。这时，该对象就仍然可以存活。

## 垃圾回收算法

就像在生活中一样，如果有许多杂乱的垃圾，而且还需要将他们进行回收处理，那无疑是一样巨大的工作。

从判定对象存活的角度出发，垃圾收集算法主要可以分为“引用计数垃圾收集”以及“追踪式垃圾收集”。下面主要介绍追踪式垃圾收集的相关理论。

### 理论支撑：

分代收集理论的理论依据就是大多数的对象都只有短暂的生命周期，而那些经历过多次垃圾回收的对象往往在后续的过程中就越难消亡。针对这两类的对象，有许多针对它们年龄特性的垃圾收集算法。同时，这个理论也说明了Java内存堆区域必须进行分类，将相应的对象放入相应的内存区域中，方便后续的回收。

通常， 常用的分代算法会大致将区域分成新生代和老年代。但垃圾收集并非简单的分代就可以解决，需要考虑分代的内存大小，各个分代适用算法，垃圾回收算法的用时和次数，跨分代对象之间的引用关系等等。

针对回收对象算法的用时和次数问题，Java虚拟机回收算法类型可以分为整堆收集(**Full GC**),部分收集(**Partial GC**),其中部分收集又分为新生代收集(**Young GC**)和老年代收集**(Major/Old GC**)。

而针对跨代引用问题，可能会出现老年代引用新生代的的情况，而这些对象却又不被GC Roots的可达性分析计算到。

> 特别说明一下，在某些虚拟机的设计里，GC Roots的可达对象得都是在年轻代的，也可以可达到老年代，但绝对只占一小部分，甚至要做到忽略不计。为何虚拟机的内存区域要有分代，不可以直接使用GC Root 可达就可以满足垃圾回收了吗？显然是因为GC Roots不满足，因为大部分的系统中的对象，都会有年龄分代的特征，如果老年代的大部分对象都挂在GC Roots下，又因为老年代对象不容易消亡，就会给GC Roots关联很对对象，显然不行。因此，GC Roots所可达的对象大部分因是在年轻代中的。

那么要如何解决老年代关联新生代的问题？那么我们就需要承认一个普遍的规律，那就是老年代里对新生代的引用及其之少。如果不如此默认，那么我们要进行年轻代的垃圾回收时，就必须考虑到老年代所引用的年轻代对象，我们需要遍历所有老年代对象，再遍历每个对象的所有引用对象，那这绝对是非常消耗时间且没有意义的(因为可能你检索了100个对象，只有一个老年代对象引用了新生代对象)。

这时候，为了记忆这些老年代引用的年轻代对象，需要加入一个记忆集，他相当于一个标志的集合，用于记忆老年代的那些内存空间有引用年轻代的对象。显然这会加大内存开销，不过就避免了遍历整个老年代。假设100个老年代对象只有一个引用新生代对象，那么假设老年代有5000个对象实体，那么也就只有50个这样的对象，那么记忆集也就只有50。也就是说，记忆集的大小取决于老年代引用新生代的概率。

### 相关算法分类

### 标记-清除算法

标记-清除算法核心就是通过GC Roots引用链，标记出堆中的垃圾对象，进而执行清除操作。该算法适用于存活对象较多，内存空间较小(一定程度上来说是该分区的内存空间小，相关的对象较少)的场景。比如老年代的垃圾回收。当然他也会造成内存碎片，执行效率低等问题。

### 标记-复制算法

标记-复制算法是解决回收大量垃圾对象所造成的执行效率低的问题。它具体是将这块内存区域细分成两块区域，其中一块区域是空的，另一块区域用于存放对象。当存放对象的那块区域空间不够了，执行标记复制，将存活的对象标记并复制到另一块区域中（通常被复制过去的对象的数量相对较少），然后将已经适用过的对象区域的对象全部清除。

这种算法适用于对象存活时间较短的区域，例如新生代区域。通常新生代的对象在一次GC之后，便只有少数对象能够存活，故可以利用这个性质对内存空间进行进一步地划分。通常是将内存区域分为较大的Eden区和两个较小的Sur[vivo](https://www.zhihu.com/search?q=vivo&search_source=Entity&hybrid_search_source=Entity&hybrid_search_extra=%7B%22sourceType%22%3A%22article%22%2C%22sourceId%22%3A%22159059531%22%7D)r(S1,S2)区域，在最开始时只使用较大的Eden区以及其中的一块Survivor区，当需要GC时，就需要将存活对象复制到另一块Survivor区域中，然后将Eden区和已经使用的Survivor区域一次清空。

所以，如果当配置Survivor区域的内存空间过高，那么就会浪费一大批的内存，而如果Survivor区域内存过小，那么每次GC存活下来的对象数量可能很多，那么在GC时的复制可能会导致Survivor空间存放不下当前存活对象，那么就会发生晋升，会导致老年代对象的增多，一定程度上也就意味着Old GC变多了。

### 标记-整理算法

标记-整理算法的思路是也是先标记出存活对象，不同于标记清除算法的是，它采取移动对象的方式来进行内存回收，为的是解决内存碎片的问题。但是由于标记整理算法需要进行移动对象，那么对象的内存地址就会变更，如果允许用户线程和GC线程同时进行的话，那么在GC移动对象之后，对象地址发生变化，而此时用户线程使用到了这个对象，但其并不知对象地址已经发生了移动，故会发生问题。为了避免这种问题，就需要在GC的时候暂停所有的用户线程，造成Stop The World的现象。
